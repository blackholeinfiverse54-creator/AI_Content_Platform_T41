## 1. Executive Summary

### What is ARTHA?

ARTHA is a contract-driven governance platform for financial services built on Node.js (ESM). It implements a capability-based architecture where 9 governance contracts define authority boundaries, API endpoints, ownership, failure behaviors, and dependency graphs. Runtime enforcement is implemented through:

- **Authority boundary middleware** (`backend/src/middleware/authorityBoundary.js`) mounted globally in `server.js`
- **Capability loader** (`backend/src/runtime/capability_loader/index.js`) that builds a runtime registry from contracts
- **Service interceptor** (`backend/src/runtime/enforcement_engine/index.js`) using ES6 Proxy to intercept service calls
- **Contract validator** (`backend/src/runtime/contract_validator/index.js`) that validates contracts at startup

## 3. Convergence Checklist

For each acceptance criterion, showing exactly HOW it is satisfied:

### 3.1 Authority Enforcement Cannot Be Bypassed

| Evidence | Source | Status |
|---|---|---|
| `authorityBoundary.js` middleware mounted globally | `backend/src/server.js:155` | `app.use(authorityEnforcement)` |
| `guardCollectionAccess` throws by default | `backend/src/middleware/authorityBoundary.js:11` | Documented in header: "it THROWS by default" |
| Service interceptor wraps all service instances | `backend/src/runtime/enforcement_engine/index.js:195-282` | `createServiceProxy()` using ES6 Proxy |
| 1292 adversarial tests pass | `evidence/adversarial-results-2026-06-29.json` | 0 failures across 20 categories |
| Runtime violation logging | `backend/src/runtime/enforcement_engine/index.js:164-177` | `logViolation()` writes to `authority-violations.jsonl` |

**How enforcement works:**

1. **HTTP Layer**: Every request passes through `authorityEnforcement` middleware (mounted at `server.js:155`), which intercepts ALL requests before any route handler executes. It loads authority definitions from `contracts/capability_contracts/*.json` at startup — not from hardcoded maps.

2. **Service Layer**: `createServiceProxy()` wraps service instances with ES6 Proxy objects that intercept every method call and verify the calling capability owns the collections being accessed. In production mode, violations throw errors; in development, they log warnings.

3. **Collection Layer**: `guardCollectionAccess()` provides explicit collection-level checks. It is called from route handlers and throws by default when no capability context exists — developers cannot forget to call it.

4. **Mongoose Layer**: `interceptCollectionAccess()` provides Mongoose model-level hooks that check authority before query execution on both reads and writes.

**How contracts drive behavior:**

The `capabilityLoader` (`capability_loader/index.js:210-258`) reads ALL contract JSON files from `contracts/capability_contracts/` at startup, normalizes them, and builds a `Map<string, object>` registry keyed by `capability_id`. This registry is:

- Used by `authorityBoundary.js` to populate `AUTHORITY_MAP` (no hardcoded values)
- Used by `enforcement_engine` to build `COLLECTION_MAP` for service proxying
- Used by `authority_runtime` to build ownership maps for service-level enforcement
- Validated by `contract_validator` which checks all required fields, semver format, ownership conflicts, and circular dependencies
